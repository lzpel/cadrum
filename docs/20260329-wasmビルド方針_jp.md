# OCCT wasm32ビルド方針

## 目標

```
make -C wasm-experiment/ generate
```

が通ること。これは以下を意味する:

1. chijin が `wasm32-unknown-unknown` ターゲットでライブラリとしてコンパイルできる
2. アプリケーション側が `chijin::Solid`, `chijin::Boolean`, `chijin::Shape` 等を普通に使える
3. wasm-bindgen でJS関数としてエクスポートできる
4. wasm-pack が .wasm + JS glue + .d.ts を生成できる

## 意義

chijin自体をwasm化するのではなく、**chijinを使ったアプリケーションが
`wasm32-unknown-unknown` でビルドできる**ことがゴール。

wasm-bindgenを付けるのはアプリケーション側の責任であり、
chijinはただのライブラリとしてwasmターゲットでリンクが通ればよい。

### wasm-experiment の構成

```
wasm-experiment/
  Cargo.toml        # chijin = { path = ".." }, wasm-bindgen = "*"
  makefile           # generate: wasm-pack build
  src/
    lib.rs           # #[wasm_bindgen] pub fn chijin_svg() -> String
    chijin.rs        # examples/chijin.rs のコピー (chijin() -> Solid)
```

アプリ側 (`src/lib.rs`):
```rust
use wasm_bindgen::prelude::*;

#[wasm_bindgen]
pub fn chijin_svg() -> String {
    chijin::chijin().to_svg(DVec3::new(1.0, 1.0, 1.0), 0.5).unwrap()
}
```

JS側:
```js
import init, { chijin_svg } from './pkg/wasm_experiment.js';
await init();
document.getElementById('svg').innerHTML = chijin_svg();
```

この例では `Solid::cylinder`, `Boolean::union`, `Boolean::subtract`,
`Shape::to_svg` が全て wasm 上で動作する必要がある。

## 現状

`src/lib.rs` に `#[cfg(not(target_arch = "wasm32"))]` ゲートがあり、
wasm時はOCCT関連モジュールが全て除外される → chijinの機能は使えない。

このゲートを外してビルドが通る状態にすることが作業のゴール。

## なぜ可能か

chijinが使うOCCTの機能は本質的にメモリ演算で閉じている:

| 機能 | OS依存 | 備考 |
|------|--------|------|
| Shape構築 (Box, Cylinder等) | なし | 純粋なメモリ演算 |
| Boolean演算 (Fuse, Cut, Common) | なし | 純粋なメモリ演算 |
| Meshing | なし | 純粋なメモリ演算 |
| STEP/BREP I/O | なし | RustReader/RustWriter経由で既にファイル不要 |
| SVG/HLR投影 | なし | 純粋なメモリ演算 |

OS依存はTKernelのOSD層に集中しており、chijinの機能パスからは到達しない。

## アーキテクチャ

```
[JS] wasm-bindgen
  ↓
[Rust] アプリケーション (wasm-experiment)
  ↓ use chijin::*
[Rust] chijin (ライブラリ)
  ↓ cxx FFI
[C++] OCCT (静的ライブラリ、OS依存部スタブ化)
  ↓
[C++] libc++ freestanding (スレッド/ファイル/locale無効)
  ↓
[C] 最小限のCシンボル (malloc/free/memcpy/abort)
  ↓ Rustランタイムが提供
wasm linear memory
```

## 3つの作業

### 作業1: libc++ freestandingビルド

LLVMの libc++ (libcxx) をwasm32ターゲットでクロスコンパイルする。

libc++はC++標準ライブラリ（`std::string`, `std::vector`等）のLLVM実装。
OS依存の処理は下層のCライブラリ（libc）に委譲する設計のため、
ファイル/スレッド/locale等を無効にすればlibc無しでビルドできる。

```
std::cerr << "hello"
  → ostream::write()
    → streambuf::overflow()
      → fwrite()           ← ここでC関数に委譲
        → (wasm: スタブ)
```

```cmake
cmake -G Ninja llvm-project/runtimes \
  -DCMAKE_C_COMPILER_TARGET=wasm32-unknown-unknown \
  -DCMAKE_CXX_COMPILER_TARGET=wasm32-unknown-unknown \
  -DLLVM_ENABLE_RUNTIMES="libcxx;libcxxabi" \
  -DLIBCXX_ENABLE_SHARED=OFF \
  -DLIBCXX_ENABLE_FILESYSTEM=OFF \
  -DLIBCXX_ENABLE_THREADS=OFF \
  -DLIBCXX_ENABLE_MONOTONIC_CLOCK=OFF \
  -DLIBCXX_ENABLE_RANDOM_DEVICE=OFF \
  -DLIBCXX_ENABLE_LOCALIZATION=OFF \
  -DLIBCXX_ENABLE_EXCEPTIONS=OFF \
  -DLIBCXXABI_ENABLE_EXCEPTIONS=OFF \
  -DLIBCXXABI_ENABLE_THREADS=OFF
```

無効化の理由:
- `FILESYSTEM=OFF` — wasmにファイルシステムなし
- `THREADS=OFF` — wasmはシングルスレッド
- `MONOTONIC_CLOCK=OFF` — `clock_gettime` なし
- `RANDOM_DEVICE=OFF` — `/dev/urandom` なし
- `LOCALIZATION=OFF` — ロケール不要、POSIX依存排除
- `EXCEPTIONS=OFF` — wasm例外は未成熟

libc++が要求するCシンボルは全てRustのwasmランタイムが提供する:
- `malloc`/`free`/`realloc` → Rustの`dlmalloc`
- `memcpy`/`memmove`/`memset`/`strlen` → `compiler-builtins` crate
- `abort` → `__builtin_trap`

現在 `wasm-sysroot/` に自作C++標準ヘッダがあるが、
これはcxxクレートのコンパイルを通すためのスタブ実装であり、
libc++ freestandingに置き換える。

### 作業2: OCCTソースのスタブ化

build.rsの既存 `patch_occt_sources` + `stub_all_top_level_bodies` を拡張。
OS依存ファイルのメソッドボディを空化する。

スタブ対象 (TKernel OSD層):

| ファイル | 依存 | 方針 |
|---------|------|------|
| `OSD_File.cxx` | open/read/write/close | 空化 |
| `OSD_Path.cxx` | ファイルパス操作 | 空化 |
| `OSD_Signal.cxx` | signal, MessageBoxA | 空化 |
| `OSD_SharedLibrary.cxx` | dlopen/LoadLibrary | 空化 |
| `OSD_Thread.cxx` | pthread_create | 空化 |
| `OSD_Environment.cxx` | getenv/setenv | 空化 |
| `OSD_Directory.cxx` | mkdir/opendir | 空化 |
| `OSD_FileSystem.cxx` | fopen/fclose | 空化 |
| `OSD_Parallel.cxx` | TBB/OpenMP | 空化 |
| `OSD_MemInfo.cxx` | /proc/meminfo, sysinfo | 空化 |
| `Standard_ErrorHandler.cxx` | setjmp/longjmp | 空化 |
| `FSD_File.cxx` | ファイルI/O | 空化 |
| `Plugin_MapOfFunctions.cxx` | 動的ロード | 空化 |

スタブ不要 (メモリ演算のみ):
- `Standard_MMgrOpt.cxx` — malloc/freeラッパー → Rustのallocatorで動作
- `NCollection_*` — コンテナ・アルゴリズム
- `gp_*`, `TopLoc_*`, `TopoDS_*` — 幾何データ構造

手順:
1. `cargo build --target wasm32-unknown-unknown` でリンクエラー収集
2. 未解決シンボルからソースファイル特定
3. `patch_occt_sources` に追加して空化
4. リンクエラーが消えるまで繰り返し

### 作業3: build.rs のwasm対応

現在のbuild.rsは `target_arch == "wasm32"` で即returnしている。
これをOCCTのCMakeビルド + スタブ化 + libc++リンクを行うように変更する。

CMake設定のwasm差分:
```cmake
-DCMAKE_SYSTEM_NAME=Generic
-DCMAKE_C_COMPILER=clang
-DCMAKE_CXX_COMPILER=clang++
-DCMAKE_C_COMPILER_TARGET=wasm32-unknown-unknown
-DCMAKE_CXX_COMPILER_TARGET=wasm32-unknown-unknown
-DCMAKE_CXX_FLAGS="-fno-exceptions -nostdinc++ -isystem <libc++/include>"
```

完了後、`src/lib.rs` の `#[cfg(not(target_arch = "wasm32"))]` ゲートを外す。

## リスクと対策

| リスク | 影響 | 対策 |
|--------|------|------|
| OCCTのスタブ化漏れ | リンクエラー | エラー駆動で追加 |
| OCCTが内部でファイルパスを仮定 | 実行時エラー | stream I/O経路のみ使用 |
| libc++のCシンボル不足 | リンクエラー | wasm-sysrootのC関数宣言を流用 |
| wasmバイナリサイズ | 数MB超 | wasm-opt + LTO + 未使用コード除去 |
| STEP readerのスキーマファイル参照 | 読み込み失敗 | STEPCAFControl_Readerの内部パスを調査 |

## 成功基準

```
make -C wasm-experiment/ generate
```

が通り、生成された .wasm をブラウザで実行して
chijin (太鼓) のSVGが描画されること。
