# SDF を CAD に使う設計考察

sandbox-sdf での一連の実験 (point_loop → edge_loop → edge_loop2) を踏まえて、「SDF を CAD に使う」という発想のメリット・デメリット・現実的なハイブリッド設計案をまとめる。

## 観測された SDF の弱点 (CAD 用途)

sandbox-sdf の実験で具体的に露呈した問題:

- **グリッド離散化精度に律速**: marching squares の RES=1024 だと cell size ≈ bbox/1024。CAD で要求される 1e-6 相対精度には届かない (Newton 射影で点の位置は SDF 精度まで上がるが、トポロジー判定は grid 精度で律速)
- **抽出の幾何曖昧性**: 例えば矩形の 4 頂点は外接円上に乗るので、Step 3 (Visvalingam) で内部点が全部消えた後の Step 4 では Line vs Circle が原理的に区別不能 (edge_loop2 で実際に観測。`out/issue_cover_v2.png` 参照)
- **位相が脆い**: 細い壁・接する曲線・cusp など、cell width 以下の特徴はサンプリングミスで連結性が変わる
- **微分の不連続**: SDF の角点で ∇d がジャンプ。5点差分ラプラシアンは EPS と cell size が同オーダーで数値死

## 観測された SDF の強み (CAD 用途)

- **Boolean が `min`/`max` 一行で済む**: 五輪の和は `c1.min(c2).min(c3).min(c4).min(c5)`
- **交点計算が要らない**: BRep ではここがバグの温床 (OCCT の BOP は edge case で頻繁に転ぶ)。SDF は「結果が物理的に正しい」ことを実装側で何もせず得られる
- **位相が自動で正しい**: 円が重なれば 1 連結成分、離れれば 2 つ、接すれば cusp — 表現側で何もしなくても結果が正しい
- **DSL として書きやすい**: 関数型 + min/max の宣言的構成。プロシージャル CAD と相性が良い

## 五輪の和の例

OCCT 側で書くと:
1. 5 円定義
2. `BRepAlgoAPI_Fuse` で union
3. 各円対の交差 (10 ペア) → 20 頂点を出す
4. 各円弧の弧長パラメータを正しく計算
5. ループを TopoDS_Wire に組む

100 行超 + degenerate case でデバッグ地獄。SDF なら 1 行。逆に SDF から CAD への抽出は、edge_loop2 の壁にぶつかる。

## ハイブリッド設計案

「単純な SDF だけで CAD」は実験通り壁にぶつかる。以下のいずれかが現実的:

### Approach A: 原始メタデータ付き SDF (primitive provenance)

評価時に `(distance, primitive_id)` のタプルを返す:

```rust
fn sdf_circle_annotated(p: DVec2, id: PrimId) -> (f64, PrimId) {
    (p.length() - 1.0, id)
}

fn min_annotated(a: (f64, PrimId), b: (f64, PrimId)) -> (f64, PrimId) {
    if a.0 < b.0 { a } else { b }
}
```

境界上の各点で「**今どの原始の距離が勝っているか**」が分かる。boundary tracing 中に primitive_id が切り替わる点が真の交点 (vertex) で、各 segment は「ある単一 primitive の境界の一部」= 線分か円弧か NURBS の弧か、解析的に決まる。

これで抽出問題が「fit する」から「**ID 境界をトレースする**」に変わる。精度は SDF 評価精度ではなく、**ID 境界検出精度** で決まる (sign の変化と同じ、marching squares レベルで十分)。

五輪の例:
- 10 原始 (5 outer arc + 5 inner arc)
- min で和を取る
- 境界をトレースすると ID が 10 回切り替わる → 10 vertex
- 各 segment は ID == k の単一円弧 → そのまま `Edge::Circle { 中心: c_k, 半径: r_k, 角度区間: [θ1, θ2] }` で厳密

edge_loop2 で起きた「矩形の内接円誤認」も発生しない。原始 ID が「これは直線」と教えてくれるから。

### Approach B: SDF を rendering 専用に割り切る

authoring は OCCT の直接 API (circles, lines, union) で書き、SDF は preview/可視化だけに使う。SDF の利点 (構成の楽さ) を諦める。今の sandbox-sdf 単独で完結するならこっち寄り。

### Approach C: SDF を AST のまま持つ (F-rep / CSG-tree)

SDF を関数として評価せず、**式木 (expression tree)** として保持。`Union(Circle(c1, r1), Circle(c2, r2))` のような構造。

- boolean は AST ノード追加
- 表示は evaluate
- BRep 化は AST traversal で OCCT API を呼ぶ

これは F-rep / CSG-tree の伝統的アプローチ。一番厳密だが、SDF の任意関数性 (`p.length().sin() * exp(-p.x)` みたいな自由関数) は犠牲。

## 結論

- 「**単純な SDF → MS → BRep**」抽出パイプラインは本質的に病的。実用は無理。
- ユーザーが感じている SDF の魅力 (boolean が楽・五輪の和を書きやすい) は本物。
- 次の一手としては **Approach A (primitive ID annotated SDF)** が最有力:
  - SDF のオーサリング上の楽さを保ったまま BRep の厳密さを得られる
  - sandbox-sdf を拡張して `sdf: Fn(DVec2) -> (f64, PrimId)` 型のパイプラインを試す
  - edge_loop の精度問題が **構造的に消える** はず (推測)
- Approach C (CSG-tree) は OCCT との整合性が綺麗。本格 CAD なら最終形はここ。
