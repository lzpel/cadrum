# `Solid::history` とは何か — 定義の明確化

## Context

[#156](https://github.com/lzpel/cadrum/issues/156) は shell / fillet / chamfer に history を実装する issue だが、その前提である「history とは何か」の定義が曖昧なまま進もうとしていた。issue 本文は history を「`Modified(face)` のみを emit する」と表現しているが、実装 (`cpp/wrapper.cpp` の `builder_cells`) を読むと**この表現は不正確**であり、この誤った定義のまま shell/fillet/chamfer を実装すると colormap remap が壊れる。

本ノートはコードと OCCT 仕様を根拠に、history の正確な定義を確定する。結論を先に書くと:

> **`Solid::history` は「結果に生き残った各 face を、同種(face)の唯一の元 face に対応付けた派生系譜 (derivation lineage)」である。identity (pass-through, 無変更面) ∪ Modified を含み、Generated (edge→face のような新規面) は含まない。「Modified のみ」ではない。**

関連: [#156](https://github.com/lzpel/cadrum/issues/156), [20260425-Solid_history導入とCxxOpaqueStruct廃止](20260425-Solid_history導入とCxxOpaqueStruct廃止.md), [20260420-OCCTトポロジ不変性と設計含意](20260420-OCCTトポロジ不変性と設計含意.md)

## 現状の実装 (根拠)

### スキーマ

`Solid::history: Vec<u64>` は flat `[post_id, src_id, post_id, src_id, ...]` の **face TShape\* ペア**。`id()` は `reinterpret_cast<uint64_t>(shape.TShape().get())` で、Face/Edge/Solid のみが持つ (`cpp/wrapper.cpp:945-955`)。公開 API は:

```rust
// src/traits.rs:461-465, src/occt/solid.rs:237-239
fn iter_history(&self) -> impl Iterator<Item = [u64; 2]> + '_;
```

history を populate するのは boolean (`builder_cells`) と `clean` (`builder_clean`) のみ。extrude / sweep / loft / bspline は入力に face を持たないので by design で空。shell / fillet / chamfer は **未実装 (TODO)** で現状空。

### 核心: relay は identity も載せる

`builder_cells` の history 採取は「Modified だけ」を見ているのではない (`cpp/wrapper.cpp:548-567`):

```cpp
static void collect_relay_mapping_cells(BOPAlgo_CellsBuilder& cb, const TopoDS_Shape& src,
                                        std::unordered_map<uint64_t, uint64_t>& relay) {
	for (TopExp_Explorer ex(src, TopAbs_FACE); ex.More(); ex.Next()) {
		const TopoDS_Shape& sf = ex.Current();
		uint64_t src_id = reinterpret_cast<uint64_t>(sf.TShape().get());
		if (cb.IsDeleted(sf)) continue;                 // 消えた face は載せない
		const NCollection_List<TopoDS_Shape>& mods = cb.Modified(sf);
		if (mods.IsEmpty()) {
			relay[src_id] = src_id;                     // ★ pass-through (無変更) も identity として載せる
		} else {
			for (... it : mods)                          // Modified は modified 先 → src を載せる
				relay[<modified tshape>] = src_id;
		}
	}
}
```

そのうえで末尾の `BRepBuilderAPI_Copy` が**全 face に新しい TShape を振り直す**ので、`emit_from_pairs` は生き残った face すべてについて `[post_id(新), src_id(元)]` を emit する (`cpp/wrapper.cpp:527-545, 628-630`)。

→ **history = 「Modified された face」ではなく「結果に生き残った全 face → その元 face」**。Modified も pass-through も等しく載る。deep-copy のため identity 面でも `post_id != src_id` になる。

### history の consumer

- **colormap remap** (`src/occt/solid.rs:470-482`): 元 face の色を結果 face に引き継ぐ。**無変更(pass-through)面の色追従にこそ identity エントリが必要**。
- **切断面の特定** (`examples/08_shell.rs`): `iter_history()` を `src ∈ cutter_face_ids` でフィルタし、cutter 由来の post_id (= 切断面) を集めて shell の開口に使う。

## 各質問への回答

### Q1 — TShape が付与されるのは Face / Edge / Vertex / Solid / Compound のどれか？他にあるか？

OCCT の `TopAbs_ShapeEnum` **8 種すべて**が `TopoDS_Shape` であり TShape を持つ:

`COMPOUND` / `COMPSOLID` / `SOLID` / `SHELL` / `FACE` / `WIRE` / `EDGE` / `VERTEX`

つまり挙げられた 5 種に加えて **SHELL・WIRE・COMPSOLID** も TShape を持つ。cadrum が Rust 型として公開し `id()` を提供するのは **Solid / Face / Edge の 3 種のみ** (`src/lib.rs`)。Vertex / Wire / Shell / Compound は内部表現で、Wire は `Vec<Edge>`、Shell/Compound は Solid/内部 CompoundShape に吸収される。

### Q2 — TShape を持つもののうち Modified / Generated で捕捉できるのは？

アルゴリズムが分割・生成する**下位 sub-shape**、すなわち **FACE / EDGE / VERTEX**。container 系 (WIRE / SHELL / SOLID / COMPOUND) は個別追跡されず、faces 等から再構築される。cadrum の history は現状 **FACE のみ**を利用している。

### Q3 — 変形を捕捉する手段は Modified / Generated の 2 つだけか？理解は正しいか？

**いいえ、3 つある。** OCCT の履歴 3 状態は **Modified / Generated / Deleted (`IsRemoved`)**。質問中の理解も次の通り修正が必要:

| 状態 | 意味 | 例 |
|---|---|---|
| `Generated(S)` | S から**新たに生まれた別 role/次元**の sub-shape。S 自身は別所に残りうる | edge を fillet → 丸み面が generate (edge→face) |
| `Modified(S)` | S が**同種**の sub-shape として継続 (trim / split / merge)。元 S は結果に現れず継続体が現れる | boolean で face が交線で分割 (face→face(s)) |
| `Deleted(S)` | S が継続体なく消滅 | 完全に削られた face |

質問の「Generated = S と別の TShape / Modified = 同一平面だが形状が変わった」は不正確。**Modified も新しい TShape を作る**ので「別 TShape か否か」では両者を区別できない。区別軸は **role / 次元**: Generated は「新規派生 (多くは次元が上がる)」、Modified は「同種継続」。

### Q4 — 中空球 (R5 球から R3 球をくり抜いた) の表面は Modified か？（実機確認済み）

**実機結果 (`Solid::sphere(5) - Solid::sphere(3)`):**

```
a faces = 1, b faces = 1
hollow faces  = 2
hollow volume = 410.501            // 523.6 - 113.1 ✓
history pairs = 2
  post=… src=…  post!=src:true  src∈A:true  src∈B:false   // 外殻 = A 由来
  post=… src=…  post!=src:true  src∈A:false src∈B:true    // 内殻 = B 由来
all src in A∪B = true,  all post != src = true,  from A = 1, from B = 1
```

**幾何的には外殻 (R5) も内殻 (R3) も変形もトリミングもされていない**（OCCT 的には `cb.Modified` が空を返す pass-through）。にもかかわらず **history には 2 件とも載る**。理由は Q3 の Modified ではなく、§現状の実装で述べた **relay の identity 分岐 + 末尾 deep-copy** によって「生き残った face は全て系譜に載る」ため。

→ **これは「history ≠ Modified」を最も端的に示す例**。質問の直感「幾何的に変形もトリミングもされないはず」は正しく、そして history はその無変更面をも (lineage として) 記録する。

### Q5 — history を「Modified（同種・境界以外の位置/形状に変化なし）」と定義するのは妥当か？

**妥当でない（狭すぎる）。** Q4 が示す通り、現実装は pass-through (純粋無変更) の face も identity として載せる。これは無変更面の色を colormap が追従するために**必要**であり、仕様上正しい。「Modified 限定」と定義すると中空球面や無変更面が history から抜け、colormap remap が壊れる。正しい定義は冒頭の通り:

> **history = 結果に生き残った各 face を、同種(face)の唯一の元 face に対応付けた「派生系譜」。identity (pass-through) ∪ Modified を含み、Generated (edge→face のような新規面) は含まない。**

`#156` の「`Modified(face)` のみ」という表現はこの意味に読み替える必要がある。shell / fillet / chamfer 実装時も、`Modified(face)` が空 (= 無変更面) の face を identity ペアとして含めて emit すべき。なお Generated 面 (フィレット円弧面・面取り面) は src が edge であり face→face スキーマに乗らないため、引き続き history に載らない（src_id 不在）— これは現スキーマの構造的制約で正しい。

### Q6 — Vertex が TShape ID を持つなら history に入れる利益はあるか？

**利益なし。**

1. cadrum は Vertex を Rust 型として公開せず `id()` も持たないので、history に Vertex を入れても**読む側 (consumer) が存在しない**。
2. history の用途はいずれも face 粒度で完結する — colormap remap は色が face 属性、切断面特定は face 単位。
3. Edge すら現状 history に入れていない。
4. AGENTS.md の「関数・構造体・依存を増やすより減らす方向を優先」に反する。

→ Vertex 追加は無 consumer の純粋なコスト増。入れない。

### Q7 — `collect_relay_mapping_cells` を template で汎用化し、shell/fillet/chamfer と共通化できるか？

**relay 採取のコア部分は共通化できる。ただし boolean だけは後段(history を吐く部分)が分岐する。** また提示された `TopAbs_FACE|TopAbs_EDGE` 走査は不可で、edge を history に含めるのは Q6 通り非推奨。

#### テンプレートが正解な理由

4 つの builder は **共通基底クラスを持たない** 2 系統に分かれる:

```
BOPAlgo_CellsBuilder         : BOPAlgo_Builder : BOPAlgo_BuilderShape          ← Modified/Generated/IsDeleted を宣言
BRepFilletAPI_MakeFillet     : BRepFilletAPI_LocalOperation : BRepBuilderAPI_MakeShape  ← こちらも宣言
BRepFilletAPI_MakeChamfer    : BRepFilletAPI_LocalOperation : BRepBuilderAPI_MakeShape
BRepOffsetAPI_MakeThickSolid : BRepOffsetAPI_MakeOffsetShape : BRepBuilderAPI_MakeShape
```

`BOPAlgo_BuilderShape` と `BRepBuilderAPI_MakeShape` は別系統だが、両方が `IsDeleted(const TopoDS_Shape&)` と `Modified(const TopoDS_Shape&) -> const TopTools_ListOfShape&` という **同一シグネチャ** を持つ。基底ポインタでは束ねられないが、template はコンパイル時 duck typing なので吸収できる。→ template を使う典型的な正当ケース。

```cpp
// builder 型に対して汎用化（走査する型は FACE 固定）
template <typename Builder>
static void collect_relay(Builder& b, const TopoDS_Shape& src,
                          std::unordered_map<uint64_t, uint64_t>& relay) {
	for (TopExp_Explorer ex(src, TopAbs_FACE); ex.More(); ex.Next()) {
		const TopoDS_Shape& sf = ex.Current();
		uint64_t src_id = reinterpret_cast<uint64_t>(sf.TShape().get());
		if (b.IsDeleted(sf)) continue;
		const TopTools_ListOfShape& mods = b.Modified(sf);
		if (mods.IsEmpty()) {
			relay[src_id] = src_id;                  // identity (pass-through)
		} else {
			for (TopTools_ListOfShape::Iterator it(mods); it.More(); it.Next())
				relay[reinterpret_cast<uint64_t>(it.Value().TShape().get())] = src_id;
		}
	}
}
```

#### なぜ boolean だけ後段が分かれるのか（核心）

**「共通化できない」のは relay 採取ではなく、relay を history に吐く “後段” だけ。その差は boolean が末尾に `BRepBuilderAPI_Copy` を持つかどうかに完全に帰着する。**

- boolean (`builder_cells`) は **multiple solid を合成**するため、結果が入力と `Handle<Geom_XXX>` を共有したまま任意順で drop されると `STATUS_HEAP_CORRUPTION` が起きる (Bug 1, `cpp/wrapper.cpp:501-503`)。これを断つため**結果を必ず `BRepBuilderAPI_Copy` で deep-copy する**。この copy は boolean にとって**省略不能**（省くとバグ再発）。
- copy の帰結: `cb.Modified(face)` が返す tshape は **copy 前** (`cb.Shape()` 内) のもの。しかし Rust に渡るのは **copy 後** (`copier.Shape()`) で、その face は**全く新しい tshape** を持つ。よって relay の key (copy 前 id) は Rust が見る id と一致しない。→ `emit_from_pairs` が pre/post を IndexedMap の index 対応で突き合わせて `[post(copy後), src]` に翻訳する一段が**必須**。
- 対して fillet/chamfer/thick_solid は **single solid 入力で copy しない**（`mk.Shape()` をそのまま返す。compound からの solid 抽出は tshape を変えない）。だから `mk.Modified(face)` の tshape が**最終結果そのもの**。relay の key がもう最終 id なので、**`emit_from_pairs` を挟まず relay を直接 flat 化**すればよい。

| builder | 末尾 copy | 理由 | history の吐き方 |
|---|---|---|---|
| boolean (`CellsBuilder`) | **あり (省略不能)** | multi-solid の Geom 共有 → drop 順で heap corruption (Bug 1) | relay は copy 前 id → `emit_from_pairs` で index 翻訳 |
| fillet / chamfer / thick_solid | なし | single-solid 入力で共有破棄不要 | relay の key が最終 id → 直接 flat 化 |

要するに **relay 採取ロジック自体は 4 者で完全に同一**（face を Modified/identity/Deleted で振り分ける部分）なので template で共有できる。分かれるのは「copy を挟む boolean だけが pre→post の index 翻訳を要する」という**末尾の一段だけ**であり、これは boolean のメモリ安全要件から来る本質的差分なので消せない。

#### 注意点

- **shell の open_faces 空パス**(`builder_thick_solid:1416-1449`)は `MakeThickSolid` ではなく手動の `MakeOffsetShape` + `MakeSolid` 再構築なので、単一 builder の `Modified()` に乗らない特殊ケース（original shell faces は identity、offset faces は `Generated` で src=edge ゆえ載らない）。ここだけ別扱い。
- **edge を含める軸は別問題で非推奨**（Q6）。`TopExp_Explorer` は単一 `TopAbs_ShapeEnum` しか取らず `TopAbs_FACE|TopAbs_EDGE` は不可（値が FACE=4, EDGE=6 で `4|6=6=EDGE`、黙って edge のみ走査するバグ）。consumer も無いので走査型は **FACE 固定**にする。汎用化はあくまで「**builder 型**」に対してのみ行う。

### Q8 — 3関数分解（実装した最終形）

Q7 の「relay 採取共有＋後段分岐」を、**relay を1本の map に混ぜず2本に分け**、結合（composition）で history を吐く3関数に分解した。これが `collect_relay_mapping_cells` + `emit_from_pairs` を置換した実装形（`cpp/wrapper.cpp`）:

| 関数 | 使う builder | 役割 |
|---|---|---|
| `relay_from_builder<Builder>` | 全4 (boolean/shell/fillet/chamfer) | `{result/pre id → src id}`。`Modified` 空=identity, 非空=分割先→src。`IsDeleted`/`Modified` は非 const なので `Builder&` |
| `relay_from_pair` | boolean のみ (copy あり) | `{post id → pre id}` を index 対応で採取 |
| `relay_into_history(relay1*, relay2*, out&)` | 全4 | `relay2==null`→`relay1` を直接 flat 化。`relay2!=null`→`relay2` のキー(post)から `post→pre→src` を**固定2ホップ**で合成 |

**bogus 回避の鍵**: `relay_into_history` が emit するのは**最外 map のキー（=本物の最終出力面）だけ**。boolean では `relay2` のキー(post)、no-copy では `relay1` のキー。両 map のキーの和を回さないので、`relay1` の中間/identity キーが bogus post として漏れない。固定1〜2ホップなので transitive closure も自己ループ停止判定も不要。

呼び出し:
- boolean: `relay_from_builder(cb, s, relay1)`×N → `BRepBuilderAPI_Copy` → `relay_from_pair(cb.Shape(), copier.Shape(), relay2)` → `relay_into_history(&relay1, &relay2, out)`。単一 solid 特殊ケースは builder が無いので `relay_from_pair` だけで post→src を得て `relay_into_history(&relay, nullptr, out)`。
- fillet/chamfer/thick_solid: `relay_from_builder(mk, solid, relay1)` → `relay_into_history(&relay1, nullptr, out)`。

**性能**: boolean は `relay2`（face 数 F の一時 hash map）と追加1パスが増えるが、すべて O(F) の hash 操作で、支配項の `CellsBuilder::Perform()` / `BRepBuilderAPI_Copy`（実ジオメトリ計算）に対して誤差。**順序**: `relay_into_history` は `unordered_map` を回るので `out_history` の並びは非決定的。既存 consumer（colormap remap, `08_shell.rs`, テスト）は全て順序非依存なので機能影響なし。

### 実装で判明した OCCT 挙動

- **fillet/chamfer は無変更面の tshape を保持し変更面を `Modified` で報告する**（OCCT 履歴コントラクト遵守）。`tests/history.rs` で cube の全6面が src として登場することを確認済み。よって no-copy の「relay 直接 flat 化」が正しく成立し、fillet/chamfer に `relay_from_pair` は不要。
- **`BRepOffsetAPI_MakeThickSolid` は user-removed の open_faces を `IsDeleted` で報告しない**。放置すると除去面の identity ペア（結果に存在しない face を指す bogus）が混入する。実装では `builder_thick_solid` で **src が open_face のペアを明示的に除外**して対処（`tests/history.rs` の shell テストで retained 5 面のみになることを確認）。
- **thick_solid の open_faces 空パス**（密閉殻、手動 `MakeOffsetShape`+`MakeSolid`）は単一 builder の `Modified` に乗らないので、保持される original shell 各面を identity で emit、offset 生成面は Generated 扱いで載せない。

## 結論

- `Solid::history` の正式定義は **「派生系譜 (identity ∪ Modified, face 限定, Generated は含まない)」**。
- `#156` の「Modified のみ」は不正確。shell/fillet/chamfer 実装でも無変更面 (identity) を含めて emit する。
- Vertex / Edge を history に加える計画は consumer 不在のため不要。
- history が face のみを扱い Generated 面を持てないのは、face→face スキーマの構造的制約として意図通り。
- relay 採取は `template <typename Builder>` で 4 builder 共通化できる（共通基底は無いが `IsDeleted`/`Modified` の signature が一致＝duck typing）。boolean だけは末尾の `BRepBuilderAPI_Copy`(Bug 1, 省略不能)のため pre→post の index 翻訳が要り、後段だけ分岐する。走査型は FACE 固定（edge は consumer 無し）。
- 実装は `relay_from_builder` / `relay_from_pair` / `relay_into_history(relay1*, relay2*, out&)` の3関数（Q8）。`#156` の shell/fillet/chamfer に配線済み。`MakeThickSolid` が removed open_face を `IsDeleted` で報告しない点だけ明示除外で対処。`tests/history.rs` で検証。
