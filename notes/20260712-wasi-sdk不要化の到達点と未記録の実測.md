# wasi-sdk 不要化の到達点と、issue にしか無かった実測

wasm 消費者から wasi-sdk（Windows で 625 MB）を不要にする取り組みは、issue #199 / #206 / #207 / #215 / #219 / #220 / #225 / #226 に分散して記録されてきた。
そのうち **notes に一行も書かれていない事実が多数あった**ため、issue を畳むにあたってここに集約する。以後この note が一次記録になる。

数字と行き止まりは**再発見コストが高い順**に並べた。フラグや手順はそのまま貼れる形で残す。

## 結論（2026-07-12 時点）

- **wasi-sdk 不要化は未達。** `build.rs` は今も `cpp/wrapper.cpp` をローカルでコンパイルし、README も wasm 消費者に wasi-sdk の Docker イメージを案内している。
- 障害は 2 つに分解できる。**(a) C++ を誰がコンパイルするか**、**(b) ランタイム `.a` をどう供給するか**。**(b) は解決済み**（prebuilt tarball に eh 版 libc++/libc++abi/libunwind/libc を同梱、#214 / #217）。残っているのは **(a) だけ**。
- (a) には長らく **`cxx` クレートという逃げられない壁**があった（後述 §2）。その壁を正面から壊す **PR #232（cxx 依存を捨てて手書き `extern "C"` FFI にする）が現在 open** で、これが現時点の本命。
- approach 2（wasm 化 clang を配って消費者側でコンパイル）は **PR #228 で end-to-end 実証済み**（open）。native clang とバイト等価の `.o` を出せている。

## 1. 現在地（HEAD の実態 — issue の記述と食い違う点）

- `build.rs` の `link_occt_libraries` は**無条件に** `cxx_build::bridge(...).compile(...)` を実行する。prebuilt FFI をリンクして skip する経路は main に無い。
- `build.rs` の `apply_compiler_flags` は **legacy EH**（`-wasm-use-legacy-eh=true`）を渡す。#233 / #234 で exnref から巻き戻した。exnref 前提で書かれた古い記述はすべて無効。
- OCCT_ROOT（`effective_root`）の中身を実際に書き換えるのは **`occt_from_prebuilt` / `bundle_runtime_libs` / `source::occt_from_source` / `download_and_extract_tar_gz` の 4 つだけ**。他は全部パス計算・読み取り・純粋関数。

## 2. `cxx.cc` ブロッカー — prebuilt FFI だけでは rustc-only に到達できない理由

**これが問題定義の中核**であり、どこにも記録が無かった。

- `cxx` クレートの `build.rs` は `src/cxx.cc`（cxxbridge1）を**無条件に** `cc` でコンパイルする。**skip する env も feature も無い。**
- `cxx` は cadrum の依存なので **cadrum の build.rs より先**にビルドされる。cadrum 側からは止められない（env すら渡せない）。
- `cxx.cc` は `<algorithm>` / `<iostream>` / `<string>` 等 **wasm 用 libc++/libc ヘッダ**を要求する → wasi-sysroot が要る。

つまり **cadrum のラッパーを prebuilt にしても、`cxx.cc` の 1 ファイルが残るので wasm 消費者は C++ ツールチェインから逃げられない。**

回避策は 3 つあり、**(c) を採る**:

- (a) `CXX=clangw` 透過 shim を噛ませて `cxx.cc` も clang.wasm に通す（#226 案A）
- (b) FFI `.a` に `cxxbridge1`（`cxx.cc.o`）も同梱し、消費者が `.cargo/config` で cxx の cc コンパイルを無効化する（重複シンボル対策が要る）
- **(c) `cxx` 依存そのものを捨てる（PR #232、open）** — 手書き `extern "C"` FFI にすれば C++ コンパイルは全部 cadrum 自身の `cc::Build` に入り、cadrum が wasm ツールチェインを完全に制御できる。`cxx` / `cxx-build` の 2 依存も同時に消える。

## 3. clang バージョン下限は 2 系統ある（原因が違う）

- **libc++ ヘッダ由来**: wasi-sdk-33 の libc++ は `__builtin_clzg`（**clang 19+**）を使う。ホスト clang **18.1.8** では `error: use of undeclared identifier '__builtin_clzg'`。wasi-sdk-33 同梱の clang は **22.1.0**。
- **EH 由来**: exnref は clang ≥20.1。ただし #234 で legacy に戻したので**現状この制約は効いていない**。

→ 「既存 clang を流用してヘッダだけ貰う」戦略は、**消費者 clang ≥19 のときだけ成立する。**

## 4. Option B の検証結果（clang 22.1.0、PR #227 ブランチ上での実測）

approach 1 は畳んだが、**測った事実は残る**。

- DL された prebuilt FFI = `libocct-8_0_0_rev3-wasm32_unknown_unknown-cadrum-0_8_12.a`（**442,928 B**）。`wrapper.o` は生成されない。
- 残る wasm 向け C++ コンパイルは **`cxx.cc` の 1 ファイルのみ**（§2）。
- 最小 cdylib 消費者 → `wasmtest.wasm`（**17.9 MB**）生成成功。
- **RUSTFLAGS で wasi-sysroot の lib を指す必要は無い。** OCCT prebuilt 同梱の libc++ / eh / c / c++abi / unwind で全部解決する。
  → **`notes/20260612-OCCTをwasmに載せてrun-cadrumを通す.md` の「`eh` の `-L` と `c++abi`/`unwind` を RUSTFLAGS に足す」という指示は、もう不要。**
- ネイティブ（x86_64-pc-windows-gnu）はアセット 404 → 従来パスにフォールバックし退行なし。
- 消費者フラグ:
  ```
  CXXFLAGS_wasm32_unknown_unknown=--target=wasm32-wasip1 --sysroot=<headers> \
    -isystem <.../eh/c++/v1> -isystem <.../wasm32-wasip1> -fwasm-exceptions
  ```

## 5. 再配布サブセット（Tier 1.5 / PR #218、未マージ）

**命題の訂正**: 目指すべきは「sysroot を**使わない**」ではなく「**消費者が wasi-sdk を install しない**」＝ cadrum が必要な小部分集合（ヘッダ + `.a`）を再配布する、である。

**ライセンス根拠**: libc++ / libunwind は **Apache-2.0 with the LLVM exception** → **再配布可**。同梱戦略の法的裏付けはここ。

### サイズ実測

| 対象 | サイズ |
|---|---|
| `-M` で削った最小ヘッダ集合 | **758 files / 6.6 MB**（libc++ eh 707 + libc 51） |
| 採用した triple 丸ごと `include/wasm32-wasip1` | **34 MB** |
| フル sysroot include（6 triples） | **168 MB** |
| eh ランタイム `.a`（#214 で prebuilt 同梱済み） | **13 MB** |
| wasi-sdk-33 フル（x86_64-windows） | **625 MB** |

### 隔離実証

- `make check-cadrum-bundle`（`clang.exe` + `./bundle` のみ使用）→ **green**（`Solid volume: 6000`）
- `make check-cadrum-bundle-isolated`（installed の `share/wasi-sysroot` を一時リネームして**隠した**うえで同ビルド）→ **green** → **リークが無いことを証明**

### include 解決の行き止まり（再現したい人向け）

- libc++ ヘッダは `include/wasm32-wasip1/eh/c++/v1/` に居る（**target 固有・eh 固有・拡張子無し**）。
- `-M` の最小集合 + `-isystem` は **include 順が変わる**ため、libc++ の `#include_next <float.h>` が `bits/float.h` 欠落で失敗する。
- `--sysroot=<minimal bundle>` は **`<algorithm>` not found** で死ぬ。wasi-sdk の eh 選択は内部 config / multilib 駆動で、ディレクトリを切り出して再配置できない。
- **動く手順**: triple を丸ごとコピー → `-nostdinc -nostdinc++` → 明示 `-isystem`。**clang の builtin resource ヘッダは最後に `-isystem` する**（`-nostdinc` がそれを切ってしまい、libc++ の `#include_next <stddef.h>` が壊れるため）。

### リンクフラグの根拠

- `libc.a` / `libm` / `libsetjmp` は **base にしか無く `eh/` に無い**。だから 2 本目の `-L` は **`-l c` を解決するためだけ**に存在する。
- `wasm32-unknown-unknown` は既定で libc をリンクしない → `-l c` の明示が要る。
- `-l c++` をリストに書かないのは、**`link-cplusplus`（`CXXSTDLIB=c++`）が自動で出すから**。

## 6. approach 2（wasm 化 clang）の到達点

### 成立根拠

compile と link は分離できる。link は **rust-lld** が行い、リンカは**コンパイラ非依存**で任意の `.a`/`.o` を食える。そして **コンパイラのホスト ≠ 出力ターゲット** — wasm 上で動く clang でも `--target=wasm32-wasip1` の wasm オブジェクトを吐ける。compile に要るのは**ヘッダだけ**（ランタイム `.a` はリンク時のみ）。

### 行き止まり: wasi-sdk の LLVM を wasm へクロスビルド → 失敗

- cross-configure は**成功**したが、LLVMSupport を **~55/2811 TU** までコンパイルして停止。
- 原因: **wasi-libc が POSIX を欠く**。`sigaction()` は `#ifdef __wasilibc_unmodified_upstream /* WASI has no signals */` の中にあり未提供。`fork` / `execve` も無い。→ `CrashRecoveryContext.cpp` で停止。
- mainline LLVM の Unix 層を wasi 向けに大規模移植する必要があると判明（PR #216）。

### 転換: emscripten 経路

**emscripten の musl は `sigaction` / `raise` / `kill` 等をシンボルとして提供する** → LLVM の POSIX 依存コードが**ソース改変なしで**コンパイルできる。wasi-libc の壁を回避できる。

### メモリは収まる（Tier-1 spike、native clang を proxy に peak RSS 実測）

| 最適化 | peak RSS | 時間 | `.o` |
|---|---:|---:|---:|
| `-O0` | **161 MB** | **2.7 s** | 978 KB |
| `-O1` | **163 MB** | **4.6 s** | 346 KB |

wasm32 の 4 GB 線形メモリに対し **~25 倍の余裕**。wasm-clang のオーバーヘッドは実測 **<2×** なので <1 GB に収まる。

### PR #223（merged）の成果と限界

- emscripten-standalone・**純 WASI（`env` import = 0）の clang.wasm（~67 MB）** を bare wasmtime 上で動かし、実 `wrapper.cpp`（OCCT + cxx glue + libc++ 込み）を **valid な wasm32-wasip1 exnref オブジェクト**にコンパイルできた。
- **`-Os` で native wasi-sdk-33 clang の出力と 113 バイト差（0.04%）＝事実上バイト等価。**
- 1 TU **~3.6 秒**（wasmtime ロード込み。native の 2.7〜4.6 s と同オーダー）。
- **限界**: 生成物は入力を `#embed` で焼き込んだ**専用機（~124 MB）**。generic ではなく、OCCT / wrapper が変われば焼き直しが要る。

### `env` import 7 個の正体

`--embed-file` **由来**だった: `_emscripten_fs_load_embedded_files`, `_wasmfs_jsimpl_{alloc,free,get_size,set_size,read,write}_file`。
`--embed-file` をやめると WasmFS が純 wasm の memory backend を使い、JS フックごと消えて **env=0** になる。

### その他

- wasmtime の exnref 対応 = **cfallin PR #11230（2025-11）**。実行には `-W exceptions=y -W function-references=y -W gc=y` が要る（node なら `--experimental-wasm-exnref`）。

## 7. PR #228 — generic clang.wasm + wasmtime bridge（open。**リポジトリに記録が無かった成果**）

### なぜ bridge を書くしかないのか

**WasmFS のバックエンドは MEMFS / NODEFS（node 専用）/ OPFS（ブラウザ）/ FETCHFS / JSFILEFS しか無く、standalone + wasmtime で host ディレクトリを surface する WASI バックエンドが存在しない。** `-sWASMFS` は `/` を memory backend に固定する。
→ **フラグでは解決不可能**。だから WASI と MEMFS を繋ぐ bridge を自分で書く。

### 設計

- clang.wasm に内蔵した C bridge が、**constructor** で raw `__wasi_*`（`fd_prestat_get` / `fd_prestat_dir_name` / `path_open` / `fd_readdir` / `fd_read`）で preopen を列挙し、MEMFS へ再帰コピーする。
- **destructor** で `.o` を raw `path_open` + `fd_write` を使って host へ書き戻す。
- **ファイル書き込みはクリーン**なので、clang.wasm の **stdout が raw バイナリの `0x00` を化けさせる**バグを回避できる（hex 往復が不要になる）。
- raw `__wasi_*` を直接呼ぶので import は `wasi_snapshot_preview1` に留まり、**env=0 を維持**できる。
- env=0 の維持には **import 属性の明示 + `-sERROR_ON_UNDEFINED_SYMBOLS=0`** が必要だった（最終: **env=0 / wasi=14**）。

### 実測

- GATE 1: 自明な `.cpp` → 妥当な host `.o`。
- GATE 2: 実 `wrapper.cpp` + OCCT + cxx glue（`ffi.rs.cc`）を `-Os` →
  **`wrapper.o` = 311,398 B、`ffi.rs.o` = 33,772 B ＝ native clang とバイト等価。**
- **1 コンパイル ~15 秒**。支配項は **OCCT 44 MB を MEMFS へ copy-in するコスト**。

### 実行形態

`build.rs` に **`wasmtime` クレートを build-dep として埋め込み、clang.wasm を in-process 実行**する。docker も外部 wasmtime も node も wasi-sdk も不要 ＝ 消費者は **rustc だけ**。

### 残タスク（GATE 3）

1. **`cxx` クレート自身の `cxx.cc`** をどう通すか → **PR #232（cxx を捨てる）で消滅する**（§2）
2. FFI `.a` の cadrum バージョン追従

### 配布上の制約

- clang.wasm **67 MB は crates.io のサイズ上限を超える** → OCCT prebuilt と同じく **GitHub Release から DL してキャッシュ**する方式になる。
- `-Oz` / `MinSizeRel` / brotli は未検討（現状 Release `-O3`・無圧縮）。

## 8. WasmFS / clang.wasm の罠（実測）

- **clang の raw バイナリ stdout は `0x00` が化ける**（テキスト出力とファイル書き込みはクリーン）。
- **大きい stdin は壊れる**（数 MB は可、数十 MB で切れる）。
- **WASI preopen（`--dir`）を見ない**（§7 の根本原因）。
