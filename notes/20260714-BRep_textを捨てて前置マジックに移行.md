# BRep text を捨てて、カラートレイラーを前置マジックに移行

`notes/20260406-BRep末尾カラートレイラー方式.md` の「棚上げ事項: ストリーム位置によるトレイラー回避」を回収した記録。当時 **棚上げした理由そのものが text 形式だった** ので、text を捨てることで解決した。

## 何を捨てて何を得たか

**捨てた**: BRep text (ASCII `BRepTools`) の I/O。`read/write_brep_text` は全レイヤーから消滅。

**改名**: `read/write_brep_binary` → `read/write_brep`。BRep は `BinTools` バイナリ1本になった。

**得た**: カラートレイラーが前置マジックになった。

```
旧: [BRep ペイロード][N x entry][u32 N][magic "CDCL"]   ← 末尾4バイトを「検索」して見つける
新: [BRep ペイロード][magic "CDCL"][u32 N][N x entry]   ← ペイロード終端の位置で「決め打ち」
```

**消えた関数は 14 個**。構造体は増減ゼロ(C++ の借用 streambuf を新設すればコピーを1回削れるが、それは構造体を1つ増やすので採らなかった)。

| レイヤー | 消えた関数 |
|---|---|
| `cpp/wrapper.{h,cpp}` | `read/write_brep_text_stream` |
| `src/occt/ffi.rs` | 同名の extern 宣言 |
| `src/occt/io.rs` | `read/write_brep_text` |
| `src/occt/io.rs` | `read/write_brep_with`(FFI 関数ポインタで text/binary を共有していた共通実装。バリアントが1つになれば存在理由が消える) |
| `src/occt/solid.rs` | `impl SolidStruct` の2メソッド |
| `src/traits.rs` | トレイト宣言の2メソッド |
| `src/lib.rs` | inherent ラッパ2個(codegen 生成物) |

`#include <BRepTools.hxx>` も1本落ちた。`BRepTools_History`(`clean` が使う)は別ヘッダなので巻き添えにならない。cargo 依存もリンクライブラリも減らない — `BinTools` は同じ `TKBRep` にいる。

## なぜ text だと前置マジックにできなかったのか

前置マジックには「BRep のパースがどこで終わったか」を Rust が知る必要がある。実測すると:

- **バイナリ**: `BinTools::Read` はペイロードをちょうど使い切る(パース可能な最小 prefix = ペイロード長、バイト単位で一致)。末尾にゴミが付いていても読める。`istringstream` に載せているので `tellg()` で終端が取れる。
- **テキスト**: `BRepTools::Read` は `operator>>` ベースで、書き出したペイロードが **空白で終わる**(末尾は `"*\n\n+1 0 "`)。最後の空白を消費したかどうかが確定せず、終端が ±1 バイト曖昧。さらに `RustReadStreambuf` が 8KB 先読みするので Rust 側の位置とも同期しない。

つまり前置マジックを阻んでいたのは text ただ一つだった。

## `tellg()` の罠

`BinTools::Read` がペイロードを最後のバイトまで読むと **eofbit が立つ**。`tellg()` は unformatted input function なので sentry が `good()` を見て failbit を立て、**-1 を返す**。トレイラー無しのファイルで必ず踏む。

`clear()` が必須。`clear()` は `rdstate` だけを書き換えて get ポインタを動かさないので、直後の `tellg()` は正しい位置を返す。

```cpp
iss.clear();
out_consumed = static_cast<size_t>(iss.tellg());
```

`out_consumed` は **成功時のみ書く**。例外時・null 時は触らない(Rust 側のローカルは初期化済みで、null なら読まない)。

## 前置マジックが消したバグ

末尾マジックは **検索** だった。ファイル末尾4バイトが偶然 `CDCL`(`43 44 43 4C`)になると、直前4バイトを個数として読み、ペイロードを `8 + N*16` バイト切り詰めて OCCT に渡していた。つまり **色を持たない正当な BRep が読めなくなる**。確率は低いがデータ依存で、回復手段がない。

前置マジックは確率を下げたのではなく **検索そのものを消した**。`buf[consumed..]` はパーサが取らなかったバイトなので、定義上ペイロードではありえない。衝突する相手が存在しない。

代わりに賭けているのは「`consumed` がペイロード終端ぴったり」という1点だけで、これはテスト可能な不変条件になった(`trailer_begins_where_the_payload_ends`: 同じ形状を色なし/色ありで書き、色ありが色なしのバイト列 + トレイラーそのものであることを確認)。「あらゆるペイロードの末尾4バイトが `CDCL` にならない」という検証不能な性質より、はるかに良い賭けだ。

## FFI の形

```rust
fn read_brep_stream(data: &[u8], out_consumed: &mut usize) -> UniquePtr<TopoDS_Shape>;
```

`RustReader` を渡すのをやめてスライスを直接渡した。**元から C++ 側が全部 `std::string` に吸い出していた**(`BinTools::Read` が seekable stream を要求するため)ので、「ストリーミングしていたものをバッファに変えた」のではない。color 経路は Rust の `buf` と C++ の `std::string` で **二重にバッファしていた** のが1回に減った。`RustReader` / `RustReadStreambuf` 自体は STEP の読み込み2経路が使い続けるので残る。

## 能力の喪失であることは明記した

ASCII `.brep` は OCCT の `DRAWEXE save` が吐く既定形式。これを cadrum に食わせると `BrepReadFailed` になる。単なる改名ではないので CHANGELOG の **Removed** に書いた。

なお text は truncate すると `BRepTools::Read` が **アクセス違反でプロセスごと落ちる**(`Standard_Failure` の catch を貫通)。前置マジック方式では C++ にペイロード+トレイラーを丸ごと渡すことになるので、text を残したままこの設計に移行するのは危険でもあった。バイナリ側は truncate しても `Err` を返す。
