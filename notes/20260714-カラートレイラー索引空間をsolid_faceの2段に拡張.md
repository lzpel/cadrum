# カラートレイラー索引空間を solid → face の2段に拡張

`notes/20260406-BRep末尾カラートレイラー方式.md` で定めた索引空間を変更した。バイト構造とマジックは据え置き、**索引の意味だけ**を変える破壊的変更。

## 何が問題だったか

solid レベルの色(STEP の `STYLED_ITEM` が `ADVANCED_FACE` ではなく `MANIFOLD_SOLID_BREP` を指すケース。商用 CAD が普通に吐く)を扱えるようにしたが、BRep トレイラーは **face 索引しかキーにできなかった**。そのため書き出し時に `resolve_face_colors` が solid の色を配下の全 face に展開していた。

- 見た目は保存される。
- しかし `colormap` の 1 エントリが N エントリに膨らみ、「solid 全体の色」という情報が消える。
- BRep を経由すると `Solid::color_solid()` が `None` になる。

## 新しい索引空間

書き出す compound に対して:

```
index 0 .. S-1     → decompose_into_solids(compound) の順の SOLID
index S .. S+F-1   → shape_faces(compound) の順の FACE
```

読み書きは `src/occt/io.rs` の `trailer_ids()` 1本を共有する。書き手はこれを反転して id → index、読み手は index で引く。両方向が同じベクタから導出されるので、片方だけズレることが原理的にない。

```rust
fn trailer_ids(shape: &ffi::TopoDS_Shape) -> Vec<u64> {
	let solids = ffi::decompose_into_solids(shape);
	let faces = ffi::shape_faces(shape);
	solids.iter().map(ffi::shape_tshape_id).chain(faces.iter().map(ffi::face_tshape_id)).collect()
}
```

solid レベルの色は「潰さずそのまま 1 エントリとして書く」。solid の色は solid 自身の TShape id をキーに持つ(`colormap[solid.id()]`)ので、face id と衝突しない。

## 破壊的変更を選んだ

face が solid の**後ろ**に来るため、face 索引が solid 数の分だけシフトする。マジック `"CDCL"` は据え置いたので、**旧仕様で書いた .brep を新リーダで読むと色が化ける**(旧 face 索引 0.. が solid 索引として解決される)。バージョニングもしない。旧ファイルが必要なら書き直す運用。

face を先に置けば旧ファイルは無傷のまま solid を末尾に追加できたが、「仕様として solid が先頭に来る方が自然」という判断で solid-first を採った。代償として **face 索引が solid 数に依存する** という不変条件が1本増えている。

## その不変条件が成立する根拠

「write → read で solid 数と explorer 順序が保存される」ことに全面的に依存する。これは成立する:

- BRep 読み込みは `BinTools::Read` (cpp/wrapper.cpp:235) / `BRepTools::Read` (:274) を呼ぶだけ。
- `try_sew_orphan_faces` による healing を**通らない**。呼び出し元は STEP 側の `read_step_stream` (:221) と `read_step_color_stream` (:2152) だけ。
- したがって BRep はトポロジーを書いた通りに復元する。solid が縫合で増えることも、face の TShape が張り替わることもない。

逆に **STEP が索引方式を採れない理由**もこれで説明がつく。STEP は healing を通すので索引がズレる。だから `read_step_color_stream` は索引ではなく **TShape id を明示的に FFI で運ぶ**。BRep と STEP で方式が違うのは、healing の有無という一点に由来する。

## ついでに消したもの

- `io::resolve_face_colors` — BRep 用の展開が不要になった。`mesh()` 用の展開だけが残り、そちらは `mesh()` 内にインライン展開した。`Mesh` は face 階層しか持たない唯一の消費者で、ここだけは潰すのが正しい。
- `io::resolve_color_trailer` — `trailer_ids()` + 2行の `filter_map` に吸収。
- `Solid::color_solid()` — `colormap().get(&id())` で足りる。
- `write_color_trailer` の `colormap` 引数 — `CompoundShape::new` が既に全 solid のマップをマージ済みなので `compound.colormap()` を直接使う。

## 落ち穂

`CompoundShape::decompose` は全 solid にマージ済み colormap の完全コピーを配る(src/occt/compound.rs:73)。したがって 1 つの solid の colormap には兄弟 solid のキーも入っている。書き出し時は `id_to_index` に無いキーが `filter_map` で落ちるので害はないが、この「余分なキー」の存在は前提として覚えておく。

検証中に見つけた別件: **テキスト BRep を途中で切って `BRepTools::Read` に渡すとアクセス違反でプロセスごと落ちる**(`Standard_Failure` の catch を貫通する)。バイナリ側は `Err` を返して落ちない。トレイラー誤検出などで payload が切り詰められると踏む。今回の変更とは独立の穴。
